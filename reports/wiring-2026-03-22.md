# Wiring Checkpoint — 2026-03-22

## Architecture Map

```
Entry Points:
  main/index.ts        → app startup, window creation, display media handler
  preload/*-preload.ts → 5 context bridges (main, toolbar, edit-modal, onboarding, thumbnail)
  renderer/*/index.ts  → 5 renderer processes

Orchestration:
  main/ipc/index.ts    → registerAllHandlers() — 12 handler modules
  main/store.ts        → config persistence (JSON file, partial merge)

Capability Modules:
  main/services/ffmpeg/       → post-process, encode, intro-outro, silence-cut, filters
  main/services/post-export.ts → silence removal + captions + chapters pipeline
  main/services/dependencies.ts → FFmpeg discovery/install
  main/services/assemblyai/   → transcription, captions, silence detection, chapters
  main/input/                 → global mouse/keyboard tracking via uiohook

Config Flow:
  AppConfig (store.ts) → IPC CONFIG_GET → preload bridge → renderer getConfig()
  Renderer saveConfig(partial) → IPC CONFIG_SAVE → store.saveConfig() → disk
```

## Findings

### 1. CRITICAL — CaptionConfig Never Persisted from UI

**Where:** `src/renderer/edit-modal/index.ts` (save handler, line 607-613) → `src/main/store.ts:117`

**What:** The edit modal saves `silenceRemoval` config on save (`saveConfig({ silenceRemoval: ... })`), but there is NO corresponding save for `CaptionConfig`. The `caption` field exists on `AppConfig`, has a default in `store.ts:63`, and is consumed by `processWithSilenceAndCaptions()` in `file.ts:61` — but there is zero UI to configure or persist it.

**Impact:** Captions are always disabled. The entire caption pipeline (`post-export.ts:276-418`) — including AssemblyAI transcription, ASS generation, FFmpeg burn-in, and the `CAPTION_PROGRESS` channel wiring — is unreachable dead code because `config.caption.enabled` is always `false`.

**Also affected:** The `ToolbarAPI.onCaptionProgress` listener (`toolbar-preload.ts:23-28`) and `CAPTION_PROGRESS` channel forwarding exist but can never fire.

### 2. CRITICAL — `audioDucking` Config Field: Defined, Defaulted, Never Used

**Where:** `src/shared/types.ts:97` (type), `src/main/store.ts:58` (default)

**What:** `OverlayConfig.audioDucking` is defined as `boolean`, has a default of `false`, but is never read by any renderer or main process code. Zero references in `src/renderer/` or `src/main/services/`.

**Impact:** Dead config field. If a user could toggle it, nothing would happen. No edit modal UI exists for it either.

### 3. HIGH — Recovery Pipeline Missing Silence/Captions/Intro-Outro

**Where:** `src/main/ipc/autosave.ts:115` vs `src/main/ipc/file.ts:53-72`

**What:** Normal save path runs the full pipeline:
1. `postProcessRecording()` (loudness, progress bar, pause transitions)
2. `processWithSilenceAndCaptions()` (silence removal, captions, chapters)
3. `applyIntroOutro()` (intro/outro clips)

Recovery path only runs step 1: `postProcessRecording(saveResult.filePath, config.overlay?.progressBar)` — missing pause timestamps, silence removal, captions, and intro/outro.

**Impact:** Recovered recordings get degraded export — no silence trimming, no captions, no intro/outro, no pause fade transitions. Silent quality loss on crash recovery.

---

### 4. HIGH — `CHAPTERS_READY` Channel: Sent but Never Received

**Where:** `src/main/services/post-export.ts:266` (sender) → nowhere

**What:** After processing chapters, `post-export.ts` sends `Channels.CHAPTERS_READY` to the toolbar window. But:
- No preload bridge exposes a listener for this channel
- `ToolbarAPI` type has no `onChaptersReady` method
- No renderer code listens for it

The chapters ARE saved to clipboard and `.chapters.txt` file (lines 243-253), so the file-based output works. But the UI notification path is dead.

**Impact:** The toolbar never shows chapter data to the user. The clipboard copy and file save still work, but the UI feedback loop is broken.

### 7. MEDIUM — `saveConfig` Shallow Merge Silently Drops Nested Sub-object Fields

**Where:** `src/main/store.ts:117-121`

**What:** `saveConfig()` does a shallow merge for `overlay` (`{ ...config.overlay, ...partial.overlay }`) but does NOT deep-merge nested sub-objects within overlay (`cameraEnhancement`, `socials`, `cursorEffect`, `progressBar`, `watermark`, `introOutro`).

Compare: `loadConfig()` (line 72-107) carefully deep-merges every nested object. But `saveConfig()` doesn't.

If a caller passes `saveConfig({ overlay: { progressBar: { enabled: true } } })`, it would replace the entire `progressBar` object, losing `position`, `color`, and `height` fields (they'd be undefined).

**Currently safe because:** The edit modal always sends complete `OverlayConfig` objects via `editModalAPI.save()`. The toolbar's `saveConfig({ overlay: { aspectRatio: ratio } } as any)` (line 363) is the risk — it sends a partial overlay that replaces all other overlay fields with `undefined` due to the shallow spread. However, the `...config.overlay` spread on line 119 saves it because the existing values are spread first, then only `aspectRatio` is overwritten.

But if any sub-object is partially sent (e.g., `{ overlay: { watermark: { enabled: true } } }`), the other watermark fields WOULD be lost.

**Impact:** Latent bug. Works today but fragile — any future caller sending partial nested objects will silently lose fields.

## Systemic Patterns

### Pattern: Planned Features Left Half-Wired
`UPLOAD_*` channels, `audioDucking` config, `COUNTDOWN_SKIP`, `BLUR_REGIONS_UPDATE`, `RECORDING_PAUSE_TIMESTAMPS` — all suggest features that were scaffolded (types, channels, defaults) but never connected. The type system doesn't catch these because the channels are `as const` strings and the config fields have defaults.

### Pattern: Recovery Path Falls Behind Normal Path
`autosave.ts` recovery runs a subset of the normal `file.ts` export pipeline. As new pipeline stages were added (silence removal, captions, intro/outro, pause transitions), the recovery path wasn't updated.

### Pattern: Config Surface Exceeds UI Surface
`AppConfig` and `OverlayConfig` expose fields (`caption`, `audioDucking`) that have no UI to configure them, making them permanently stuck at defaults. The config persistence layer faithfully stores and loads these fields, but they're inert.
