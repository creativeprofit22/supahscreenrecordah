## Traces

### 1. IPC Channel Coverage ✅ CLEAN

Every channel defined in `shared/channels.ts` (33 channels) has a corresponding handler registered in the main process and a preload bridge exposing it to the appropriate renderer.

One channel — `ONBOARDING_RESIZE` — has a handler (`ipc/onboarding.ts:72`) but **no preload bridge** and **no renderer usage**. See Gap #1.

### 2. PreviewSelection Type vs Actual Payload ⚠️ GAP

**What's defined** (`shared/types.ts:40-44`):
```ts
interface PreviewSelection {
  screenSourceId: string;
  screenIsBrowser: string | null;
  cameraDeviceId: string | null;
}
```

**What the toolbar actually sends** (`renderer/toolbar/index.ts:48-54`):
```ts
{
  screenSourceId, screenIsBrowser,
  cameraDeviceId,
  micDeviceId,   // ← NOT in PreviewSelection
  layout,        // ← NOT in PreviewSelection
}
```

**What the main renderer expects** (`renderer/main/preview.ts:298-304`):
```ts
interface PreviewUpdatePayload {
  screenSourceId: string;
  screenIsBrowser: string | null;
  cameraDeviceId: string | null;
  micDeviceId?: string | null;   // expects it
  layout?: string;               // expects it
}
```

**Impact**: Works at runtime because IPC passes raw objects, but TypeScript's safety is bypassed. The toolbar casts with `as Parameters<...>` to silence the error. If someone changes `PreviewSelection`, the extra fields could silently break with no compiler warning.

### 3. Onboarding: Orphaned Email Activation Step ⚠️ GAP

**HTML** (`pages/onboarding.html:46-77`): Has 3 step dots (email → permissions → prerequisites) and a full `#step-email` div with activation form, email input, and submit button.

**Renderer JS** (`renderer/onboarding/index.ts`): Only knows about 2 steps:
```ts
type StepName = 'permissions' | 'prerequisites';
const STEP_ORDER: StepName[] = ['permissions', 'prerequisites'];
```

**Result**:
- The email step HTML is **never hidden** by JS (not in `STEP_ELEMENTS`), so it renders alongside the permissions step simultaneously on load.
- 3 step dots in HTML but JS indexes only 2 steps — dot highlighting is misaligned.
- `#activation-form` has no submit handler — dead UI.
- Root-level `onboarding.html` is a stale duplicate referencing non-existent scripts (`dist/renderer/pep.js`, `dist/renderer/shim.js`, `dist/renderer/onboarding.js`).

**Action**: Remove the email step from `pages/onboarding.html`, remove the third step dot, and delete the root-level `onboarding.html` duplicate.

### 8. Recording Pipeline ✅ CLEAN

Toolbar `startRecording({screenSourceId, cameraDeviceId, micDeviceId})` → IPC handler extracts `options.micDeviceId` → forwards to main window via `MAIN_RECORDING_START` → renderer `startRecording(micDeviceId)`.

The `screenSourceId` and `cameraDeviceId` are **not** re-sent at recording start — they were already configured via `PREVIEW_UPDATE`. This is correct; the recording uses whatever streams are already active in the preview.

### 9. FFmpeg Post-Processing ✅ CLEAN

`FILE_SAVE_RECORDING` → writes buffer → `postProcessRecording()` → two-pass loudnorm with graceful fallback chain (two-pass → single-pass → simple remux → give up). All filter constants (`VOICE_ENHANCE_FILTER_BASE`, `LOUDNORM_*`, `POST_BOOST_FILTERS`, `VIDEO_ENCODE_FLAGS`) are correctly consumed. No config dropped.

### 10. Config Persistence ✅ CLEAN (minor note)

`saveConfig()` does a **shallow** merge of `overlay` (`{...config.overlay, ...partial.overlay}`), while `loadConfig()` does a **deep** merge (individually merges `cameraEnhancement` and `socials`). This asymmetry is harmless because the edit modal always sends the complete `OverlayConfig` object — but if a caller ever sends a partial overlay with only some `cameraEnhancement` fields, `saveConfig` would clobber the rest. Not a bug today, but a latent trap.

### 11. screenIsBrowser Type Mismatch ⚠️ MINOR

`ScreenSource.isBrowser` is `boolean`. `PreviewSelection.screenIsBrowser` is `string | null`. The toolbar converts: `selectedSource?.isBrowser ? 'true' : null`. The renderer checks: `!!selection.screenIsBrowser`. Works, but the type conversion is unnecessary indirection.

## Frontend ↔ Backend Wiring

### 12. CSS Build Step Missing ⚠️ **CRITICAL**

All four `pages/*.html` files reference CSS at `../dist/renderer/styles/*.css`:
- `pages/index.html` → `../dist/renderer/styles/main.css`
- `pages/toolbar.html` → `../dist/renderer/styles/toolbar.css`
- `pages/edit-modal.html` → `../dist/renderer/styles/edit-modal.css`
- `pages/onboarding.html` → `../dist/renderer/styles/onboarding.css`

The CSS source files exist at `src/renderer/styles/` (plain CSS, no preprocessing needed). But **no build step copies them to `dist/renderer/styles/`**:
- `build:renderer` only runs `tsdown` on `.ts` files
- `build:main` and `build:preload` only run `tsc`
- No copy/postbuild script exists

The CSS was committed to `dist/` once (visible in git history) but `dist/` is in `.gitignore`. After any clean clone or `dist/` cleanup, **all windows render unstyled**.

**Action**: Add a CSS copy step to `build:renderer` (or a separate `build:css` script):
```bash
cp -r src/renderer/styles dist/renderer/styles
```

## Systemic Patterns

1. **Missing build plumbing for non-TS assets**: The CSS copy step was never added when migrating from the old build system (root HTML + individual script tags) to the current one (pages/ + tsdown IIFE bundles). The old root HTML files are the smoking gun — they show a prior system where CSS was presumably in `dist/` from a different tool.

2. **Orphaned artifacts from removed features**: The email activation step, `EditModalConfig` duplicate type, `LayoutType`, `ActionType`/`ActionLabel`, `PipelineProfile`/`Feature`/`PerfFeature`, 4 root HTML files — all point to features/systems that were replaced, leaving dead code behind.

3. **Type definitions lag behind actual payloads**: `PreviewSelection` is the canonical type but the actual IPC payload has grown past it. The renderer created its own `PreviewUpdatePayload` to compensate. This is the "wrapper drift" pattern — the type became a stale subset of reality.
